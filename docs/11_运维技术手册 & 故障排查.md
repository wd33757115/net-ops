
## 4. 常见故障

### 4.1 前端 401 / 无法登录

| 检查项 | 处理 |
|--------|------|
| Django 是否运行 | 启动 daphne :8001 |
| `JWT_SECRET_KEY` 不一致 | FastAPI 与 Django 必须相同 |
| 用户不存在 | `manage.py migrate` + 演示账号或 createsuperuser |
| Token 过期 | 调用 `/api/auth/refresh/` |

### 4.2 BFF 503 / upstream 错误

| 检查项 | 处理 |
|--------|------|
| FastAPI 未启动 | 启动 uvicorn :8000 |
| `FASTAPI_BASE_URL` 错误 | 改为 `http://localhost:8000` |
| `ENFORCE_BFF_ORIGIN` | 必须经 BFF 代理，勿裸调 FastAPI（生产） |

**日志：** `django.log` 中 upstream connection refused

### 4.3 聊天无回复 / SSE 超时

**现象：** 120s/170s 超时，`chat_stream_timeout`

**排查：**

```
1. fastapi.log → supervisor 路由（workflow / skill / RAG）
2. DEEPSEEK_API_KEY 是否有效
3. Langfuse → 卡在哪个节点
4. 若 workflow_starter → celery.log → workflow_step_dispatched
5. PG: SELECT status FROM netops_workflow_runs WHERE id=...
```

**常见原因：**

- LLM API 密钥无效或网络不通
- Celery Worker 未启动
- Skill subprocess  hang（查 celery.log）

### 4.4 Workflow 步骤失败

**排查：**

```powershell
Select-String -Path .runtime\test\logs\celery.log -Pattern "workflow_step_failed"
```

```sql
SELECT step_name, skill_name, status, error_message
FROM netops_workflow_steps
WHERE run_id = '<run_id>'
ORDER BY step_index;
```

**常见原因：**

- Skill 脚本缺少 ticket_id 等必填参数
- MinIO 不可达（产物上传失败）
- subprocess 非零退出（看 celery.log stderr）

### 4.5 Skill 执行失败

| code | 含义 | 处理 |
|------|------|------|
| `skill_not_found` | 未注册 | `POST /api/skills/reload-all/` |
| `skill_disabled` | 已禁用 | toggle 启用 |
| `skill_execution_failed` | 脚本失败 | 查 celery.log、`test-run` API |
| `skill_timeout` | 超时 | 调大 `CELERY_TASK_TIME_LIMIT` |

### 4.6 RAG 无结果 / 知识问答空

| 检查项 | 处理 |
|--------|------|
| `knowledge_base/` 是否有文档 | 添加 .md/.pdf 等 |
| Chroma 索引 | 删 `vectorstore/chroma_db/` 后 reindex |
| diagnostics 中 rag=false | 查 fastapi 启动 `rag_service_*` 日志 |
| 误路由到 Skill | 调 `SEMANTIC_SKILL_MIN_CONFIDENCE` |

### 4.7 Celery 任务不消费

| 检查项 | 处理 |
|--------|------|
| Worker 进程 | 启动 celery worker |
| Windows | 必须 `-P solo`（`celery_app.py` 已默认 solo，PyCharm 手动启动也需加） |
| 队列监听 | `-Q netops.default,netops.firewall,netops.device` |
| Broker URL | `.env` 与 RabbitMQ/Redis 一致 |
| 队列堆积 | RabbitMQ UI 查看 queue |

**Windows `PermissionError: [WinError 5] 拒绝访问` / `SpawnPoolWorker-*`**

根因：未使用 `solo` pool 时，billiard prefork 在 Windows 上无法正确访问进程间信号量。

```powershell
# 1. 结束残留 Worker（任务管理器搜 python/celery，或）
.\scripts\test\stop.ps1

# 2. 正确启动（推荐用 start.ps1，或手动）
.\venv\Scripts\python.exe -m celery -A src.core.celery_tasks.celery_app worker `
  --loglevel=info -P solo -Q netops.default,netops.firewall,netops.device
```

若仍报错：确认无多个 Celery 实例争抢同一 broker；杀毒/Defender 未拦截 Python 子进程。

### 4.8 MinIO / 网盘上传失败

| 检查项 | 处理 |
|--------|------|
| MinIO 容器 | `docker compose ps minio` |
| 凭据 | `MINIO_ACCESS_KEY/SECRET_KEY` |
| 配额 | `STORAGE_MAX_FILE_BYTES` |
| code | `storage_error` |

### 4.9 Langfuse 无 Trace

| 检查项 | 处理 |
|--------|------|
| 环境变量 | `LANGFUSE_*` 三件套 |
| 降级行为 | 未配置时功能正常，仅无 UI |
| 嵌套 trace | 聊天下 Workflow 应在同一 trace 下 |

### 4.10 PostgreSQL 连接失败

| 检查项 | 处理 |
|--------|------|
| 容器状态 | `docker compose ps postgres` |
| 端口占用 | 5432 |
| 凭据 | `.env` POSTGRES_* |
| 表未建 | 重启 FastAPI 触发 create_all |

## 5. 日志排障命令

```powershell
# FastAPI 最近错误
Get-Content .runtime\test\logs\fastapi.log -Tail 100 | Select-String "error|failed|exception" -CaseSensitive:$false

# 跟踪 Celery
Get-Content .runtime\test\logs\celery.log -Wait -Tail 30

# 按 run_id 搜索
Select-String -Path .runtime\test\logs\*.log -Pattern "<run_id>"

# 按 trace_id 搜索
Select-String -Path .runtime\test\logs\fastapi.log -Pattern "<trace_id>"
```

Docker：

```bash
docker compose -f deployment/docker-compose.yml logs fastapi --tail 100
docker compose -f deployment/docker-compose.yml logs celery --tail 100
```

## 10. 应急处理

| 场景 | 动作 |
|------|------|
| FastAPI 内存泄漏 | 重启 uvicorn / compose restart fastapi |
| Celery 僵死 | restart worker；检查 solo pool |
| 队列无限堆积 | 暂停入口 → 扩容 Worker → 清理死信 |
| DB 满 | 归档 audit_logs、workflow 历史 |
| LLM 密钥泄露 | 轮换 KEY → 重启所有进程 |
