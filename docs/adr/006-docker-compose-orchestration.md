
# ADR 006: 采用 Docker Compose 进行完整服务编排

## 状态
Accepted

## 日期
2026-05-24

## 上下文
在早期版本中，我们使用单独的 Docker 容器和 PowerShell 脚本来启动应用服务。
随着架构的发展（添加了 Django 和 React），需要更一致和可维护的部署方案。

## 决策
我们决定使用完整的 Docker Compose 配置来编排所有服务：

### 完整服务列表
1. **PostgreSQL** - 主数据库（5432 端口）
2. **Redis** - 缓存和会话存储（6379 端口）
3. **RabbitMQ** - 消息队列（5672 端口）和管理界面（15672 端口）
4. **MinIO** - 对象存储（9000 端口）和控制台（9001 端口）
5. **Qdrant** - 向量数据库（6333 端口）
6. **Langfuse** - LLM 观察平台（3001 端口）
7. **FastAPI** - 核心网关（8000 端口）
8. **Django** - 后端 API（8001 端口）
9. **React** - 前端应用（3000 端口）

### 配置策略
- 所有服务定义在单个 `docker-compose.yml` 文件中
- 环境变量统一从项目根目录的 `.env` 文件加载
- 使用 Docker volumes 进行数据持久化
- 服务依赖关系明确指定

## 理由
1. **一致性** - 所有开发者和部署环境使用完全相同的配置
2. **可维护性** - 单一的配置文件，易于理解和修改
3. **一键启动** - `docker compose up -d` 即可启动所有服务
4. **隔离性** - 各服务运行在独立容器中，环境干净
5. **扩展友好** - 易于添加新服务或修改现有服务

## 后果
- 需要维护更大的 docker-compose.yml 文件
- 启动时间可能稍长（多个容器同时启动）
- 系统资源需求增加
- 部署体验显著改善
- 新人上手更容易
