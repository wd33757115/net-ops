
# ADR 005: 采用 Django + React 作为主要前端架构

## 状态
Accepted

## 日期
2026-05-24

## 上下文
在早期版本中，我们使用 Streamlit 作为快速原型的前端方案，它提供了开箱即用的 Web 界面和良好的 Python 集成。
然而，随着项目的发展，我们发现 Streamlit 存在以下限制：
1. 输入框位置难以控制，无法实现专业的聊天界面布局
2. 定制化能力有限，难以实现复杂的 UI 交互
3. 性能问题，页面加载和刷新较慢
4. 不支持现代化的前端架构和组件复用

## 决策
我们决定采用以下技术栈作为主要的前端和中间层架构：
1. **React** - 用于构建现代化的聊天界面（3000 端口）
2. **Django** - 用于 API 代理和用户管理（8001 端口）
3. **FastAPI** - 保持作为核心网关（8000 端口）

### 架构设计
```
用户 &lt;--&gt; React (3000) &lt;--&gt; Django (8001) &lt;--&gt; FastAPI (8000) &lt;--&gt; 核心服务
```

### 职责划分
- **React 前端** - 提供专业的聊天 UI，包括消息展示、输入区域、侧边栏等
- **Django 后端** - 处理用户请求，转发到 FastAPI，提供会话管理
- **FastAPI** - 保持不变，继续作为核心业务网关

## 理由
1. **更好的用户体验** - React 可以实现类似 ChatGPT、豆包的专业聊天界面
2. **灵活的定制** - 完全的控制权，可以精确控制 UI 组件
3. **性能优化** - React 的虚拟 DOM 提供更好的渲染性能
4. **生态系统** - 丰富的组件库和工具支持
5. **渐进式迁移** - 可以保持 Streamlit 作为备用方案

## 后果
- 需要维护多个前端入口
- 增加了开发和部署复杂度
- 需要学习 React 相关技术
- 用户体验显著提升
