## 1. 系统概述

### 1.1 系统简介
NetOps 运维智能体是一个基于大语言模型的智能运维助手系统，采用多 Agent 架构设计，支持：

| 功能模块 | 说明 | 状态 |
|---------|------|------|
| **知识库问答** | 基于 RAG 的智能故障诊断 | ✅ 已完成 |
| **防火墙策略生成** | 自动化安全策略配置 | ✅ 已完成 |
| **网络设备管理** | 配置备份、设备巡检 | ✅ 已完成 |
| **设备分组管理** | 设备分组列表查询 | ✅ 已完成 |

### 1.2 核心特性
- **智能路由**：Supervisor Agent 根据用户意图自动路由到对应技能
- **异步执行**：Celery 异步任务处理，保证系统响应性
- **参数自动补全**：缺失参数自动使用默认值或上下文信息
- **故障兜底**：技能执行失败自动回退到 RAG 问答系统
- **超时保护**：LLM 调用超时自动触发关键词快速路由

## 3. 组件选型

### 3.1 核心技术栈

| 分类 | 组件 | 版本 | 用途 |
|------|------|------|------|
| **框架** | FastAPI | 0.104+ | API 网关 |
| **异步任务** | Celery | 5.3+ | 异步任务处理 |
| **消息队列** | RabbitMQ | 3.12+ | 任务队列 |
| **缓存/结果** | Redis | 7.2+ | 状态存储 |
| **向量存储** | Chroma | 0.4+ | RAG 向量数据库 |
| **文件存储** | MinIO | 2024+ | 对象存储 |
| **UI** | Streamlit | 1.30+ | Web 界面 |
| **LLM** | DeepSeek | API | 大语言模型 |
| **工作流** | LangGraph | 0.1+ | Agent 工作流 |
| **RAG** | LlamaIndex | 0.10+ | 检索增强生成 |

### 3.2 关键设计决策

| 决策 | 方案 | 理由 |
|------|------|------|
| **Agent 协调** | LangGraph | 支持复杂工作流编排，状态管理 |
| **异步任务** | Celery + RabbitMQ | 成熟稳定，支持任务重试和结果存储 |
| **向量数据库** | Chroma | 轻量级，易于部署，支持多模态 |
| **参数传递** | Pydantic 模型 | 类型安全，自动验证 |
| **超时处理** | 双层超时 + 关键词路由 | 保证系统响应性，防止无限等待 |

## 4. API 流转

### 4.1 核心 API 接口

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 统一聊天 | POST | `/api/v1/chat` | REST 聊天接口 |
| WebSocket 聊天 | WS | `/ws/v1/chat` | 实时流式聊天 |
| RAG 检索 | POST | `/api/v1/rag/search` | 知识库检索 |
| ITSM Webhook | POST | `/api/v1/itsm/webhook` | 通用事件接收 |
| 防火墙策略 | POST | `/api/v1/itsm/webhook/firewall-policy` | 策略生成请求 |
| 任务状态 | GET | `/api/v1/tasks/{task_id}` | 查询任务状态 |
| 文件上传 | POST | `/api/v1/chat/upload` | 文件上传 |
| 健康检查 | GET | `/health` | 服务状态检查 |

### 4.2 请求流转流程

```
用户请求 → FastAPI → Supervisor Agent → 技能路由 → 执行 → 返回结果
              │              │              │
              │              ▼              │
              │       LLM 决策 + 参数合并    │
              │              │              │
              │              ▼              │
              │       技能执行节点           │
              │              │              │
              │              ▼              │
              │       Celery 异步任务        │
              │              │              │
              └──────────────┴──────────────┘
                     结果返回
```

**聊天请求**：

## 5. 技能系统设计

### 5.1 技能注册机制

技能通过 `SkillRegistry` 统一管理：

```python
# 技能加载流程
load_all_skills() → 扫描 skills/examples/ → 注册到 skill_registry
```

### 5.2 技能执行流程

```
SkillDecision → SkillRegistry.execute_sync() → SkillBase.invoke() → Celery.delay()
                                                                       │
                                                                       ▼
                                                           任务执行完成 → 返回 SkillResult
```

### 5.3 技能参数处理

| 参数 | 来源优先级 | 默认值处理 |
|------|-----------|-----------|
| ticket_id | 用户请求 → 上下文 → 自动生成 | `POLICY_{thread_id}` |
| ticket_title | 用户请求 → 上下文 | "防火墙策略生成" |
| policy_file_url | 用户上传 → 默认测试文件 | `tools/firewall-policy/policies.xlsx` |

### 5.4 可用技能列表

| 技能名称 | 功能 | 参数 |
|---------|------|------|
| `firewall_policy` | 生成防火墙策略 | ticket_id, ticket_title, policy_file_url |
| `device_backup` | 设备配置备份 | device_name, ip_address, group_name |
| `device_patrol` | 设备巡检 | device_name, ip_address, group_name |
| `device_list` | 设备/分组列表 | list_type, group_name |

## 10. 常见问题

### Q1: 任务一直处于 "pending" 状态
**原因**：Celery Worker 未启动或未连接到 RabbitMQ

**解决方法**：
```powershell
# 检查 Docker 服务
cd deployment
docker-compose ps

# 检查 Celery Worker 日志
# 查看对应终端窗口
```

### Q2: FastAPI 服务无法访问
**原因**：端口被占用或服务未启动

**解决方法**：
```powershell
# 检查端口占用
Get-NetTCPConnection -LocalPort 8000

# 重启服务
.\stop_all.ps1
.\start_all.ps1
```

### Q3: 设备管理功能报错
**原因**：缺少 netmiko 库

**解决方法**：
```powershell
.\venv\Scripts\pip.exe install netmiko
```

### Q4: 参数缺失错误
**原因**：技能调用时缺少必需参数

**解决方法**：
- 确保传入 ticket_id 参数
- 或等待系统使用默认值自动补全

### Q5: LLM 调用超时
**原因**：网络延迟或 LLM 服务响应慢

**解决方法**：
- 系统会自动触发关键词快速路由
- 检查网络连接状态
