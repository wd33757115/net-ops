## 1. 产品概要

NetOps Agent 面向网络运维场景，通过大模型理解用户意图，自动调度 **Skill（技能）**、**Workflow（多步流程）** 与 **RAG 知识库**，完成设备备份、巡检、防火墙策略生成、配置对比、日志分析等任务。

**内置 Skill（6 个）：**

| Skill | 分类 | 触发示例 |
|-------|------|----------|
| `device-backup` | network | 备份设备配置 |
| `device-patrol` | network | 执行巡检 |
| `firewall-policy-generator` | security | 生成防火墙策略 |
| `config-diff-tool` | network | 对比配置 |
| `log-analyzer` | network | 分析日志 |
| `network-topology-analyzer` | network | 网络拓扑分析 |

## 2. Supervisor v2 详细设计

**入口：** `src/agents/supervisor/graph_v2.py` → `compiled_graph_v2()`

### 2.1 状态（SupervisorStateV2）

关键字段：

| 字段 | 说明 |
|------|------|
| `messages` | LangChain 消息列表 |
| `skill_matches` | 语义路由匹配结果 |
| `execution_plan` | LLM 结构化 ExecutionPlan |
| `intermediate_results` | 多 Skill 协同中间结果 |
| `workflow_type` / `workflow_run_id` | Workflow 插件触发 |
| `langfuse_parent_trace_id` | SSE 聊天 trace，供 Workflow 嵌套 |
| `user_id` / `user_role` / `thread_id` | 用户与会话上下文 |

### 2.2 节点流程

```
pre_process_node
  ├─ Skill 语义路由（top_k=5）
  ├─ Progressive Disclosure 加载 SKILL 指令
  └─ 新一轮用户消息时重置 intermediate_results

supervisor_node_v2
  ├─ LLM 生成 ExecutionPlan（function calling）
  ├─ 知识类问句 → next_agent=knowledge_qa
  ├─ Workflow 触发词 → next_agent=workflow_starter
  └─ 运维动作 → next_agent=orchestrator

orchestrator_node
  └─ Send fan-out（parallel / sequential）

skill_executor_v2_node
  └─ 调用 skill_registry.execute → Celery 或同步

workflow_starter_node
  └─ WorkflowEngine.start(..., parent_trace_id=langfuse_parent_trace_id)

final_aggregator_node
  └─ Map-Reduce 合并多 Skill 结果

knowledge_qa_node
  └─ Chroma RAG + LLM
```

### 2.3 路由策略

- **触发词 / 语义匹配**：`SEMANTIC_SKILL_MIN_CONFIDENCE`（默认 0.72）以下不加载 Skill
- **知识问句过滤**：避免「怎么办」类问题误匹配 device-backup
- **容错**：Skill 失败可 `fallback_to_rag`（SKILL.md 配置）

## 3. Skill 系统详细设计

**核心模块：** `src/skill_system/`

| 模块 | 文件 | 职责 |
|------|------|------|
| 元数据 | `metadata.py` | 解析 SKILL.md frontmatter |

**SKILL.md v2.0 格式：** 见 [SKILL_CREATION_GUIDE.md](./SKILL_CREATION_GUIDE.md)

**执行路径：**

```
execute_skill(skill_name, params)
  → strip 观测字段（langfuse_trace_id 等）
  → subprocess run.py 或 legacy Celery task
  → MinIO 上传（如有 artifacts）
  → record_skill_execution_span（Langfuse）
```

## 4. Workflow 详细设计

### 4.1 插件结构

```
src/workflows/{category}/{plugin_id}/
├── WORKFLOW.yaml      # 步骤定义、表达式、on_complete
├── CHAT.intent.yaml   # 聊天意图与上下文映射（可选）
└── README.md
```

示例：`src/workflows/itsm/1/`

### 4.2 引擎（WorkflowEngine）

**文件：** `src/core/workflows/engine.py`

| 方法 | 说明 |
|------|------|
| `start()` | 创建 run、Langfuse trace/span、写入 steps、dispatch 首步 |
| `dispatch_step()` | Celery 异步执行单步 |
| `handle_step_complete()` | 更新步骤状态、记录 Langfuse step span、推进下一步 |
| `build_step_params()` | 解析 WORKFLOW inputs 表达式，注入 trace 上下文 |
| `ensure_wf_trace()` | Celery Worker 恢复 Langfuse 观测上下文 |

**Run 状态：** `pending` → `running` → `completed` / `failed`

**并行步骤：** `parallel_group` 相同且连续 → 批量 dispatch

### 4.3 进度通知

- **Redis pub/sub**：`publish_workflow_event` → 前端 SSE `events/stream`
- **站内通知**：`netops_notifications`（步骤完成、Workflow 结束）

### 4.4 插件治理

- 文件系统为运行时真相源
- PG 表 `netops_workflow_plugins` / `_plugin_versions` 记录 draft/review/published 状态
- 模板市场：`netops_workflow_market_templates`

## 9. 关键设计原则

1. **Metadata 常驻** — 仅 name + description 常驻内存
2. **Progressive Disclosure** — 匹配 Skill 才注入正文（800–1500 tokens）
3. **指令注入而非函数调用** — Skill = 专业规则文档
4. **文件系统驱动** — Skill/Workflow 可 Git 版本管理
5. **3 层容错** — 加载失败降级、路由失败 RAG 兜底、执行异常 RAG 兜底
